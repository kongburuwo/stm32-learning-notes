# STM32 工程模板目录说明

本文档说明工程模板中各目录的作用与结构，方便后续开发与维护。

---

## 目录总览

| 文件夹名称 | 核心作用 | 通俗理解 | 典型内容 |
| :--- | :--- | :--- | :--- |
| **Drivers** | 存放与硬件相关的驱动层文件 | 所有和芯片/外设打交道的底层驱动，是工程的“硬件基础” | `BSP/`板级驱动、`CMSIS/`内核抽象层、`HAL_Driver/`官方HAL库、`SYSTEM/`延时/串口等基础驱动 |
| **Middlewares** | 存放中间层文件 | 第三方库、中间件，是工程的“扩展能力层” | FreeRTOS、FatFS文件系统、USB协议栈、LwIP网络协议栈 |
| **Output** | 存放工程编译输出文件 | 编译器生成的中间文件与最终产物，是工程的“产物目录” | `.o`目标文件、`.axf`调试文件、`.hex`/`.bin`烧录文件 |
| **Projects** | 存放MDK工程文件 | Keil MDK的工程配置与入口，包含编译选项、链接脚本 | `.uvprojx`工程文件、`.uvoptx`配置文件 |
| **User** | 存放用户代码与配置文件 | 业务逻辑所在的应用层，与芯片无关 | `main.c`主函数、中断服务文件、HAL库配置文件 |

---

## 各目录详细说明

### 1. Drivers 目录
- **核心定位**：硬件驱动层，是软件与硬件之间的桥梁。
- **子目录与作用**：
  - `BSP/`：板级支持包，存放LED、按键、传感器等外设的驱动文件。
  - `CMSIS/`：ARM Cortex-M 内核抽象层，包含启动文件、系统时钟配置等基础文件。
  - `STM32F1xx_HAL_Driver/`：ST官方提供的HAL库驱动文件，包含所有外设的通用驱动接口。
  - `SYSTEM/`：常用基础驱动，如延时函数、串口打印函数、系统滴答定时器等。

### 2. Middlewares 目录
- **核心定位**：中间件层，提供扩展功能的第三方库。
- **典型用途**：
  - 操作系统：FreeRTOS、RT-Thread
  - 文件系统：FatFS
  - 通信协议栈：LwIP（TCP/IP）、USB协议栈
  - 第三方算法库

### 3. Output 目录
- **核心定位**：编译产物目录，存放Keil生成的所有临时文件与可执行文件。
- **常见文件**：
  - `.o`：目标文件，每个C文件编译后的机器码
  - `.axf`：带调试信息的可执行文件，用于在线调试
  - `.hex`/`.bin`：可直接烧录到芯片的固件文件
- **开发建议**：提交到Git时，可将此目录加入`.gitignore`，避免上传临时文件。

### 4. Projects 目录
- **核心定位**：工程配置目录，存放Keil MDK的工程文件。
- **典型文件**：
  - `.uvprojx`：Keil工程主文件，包含工程结构、编译选项、包含路径等配置
  - `.uvoptx`：工程的本地配置文件，如窗口布局、调试器设置等
- **开发建议**：仅`.uvprojx`文件需提交，`.uvoptx`可加入`.gitignore`。

### 5. User 目录
- **核心定位**：用户应用层，存放开发者编写的业务逻辑代码。
- **典型文件**：
  - `main.c`：程序入口文件，包含主函数与主循环
  - `stm32fxxx_it.c/h`：中断服务函数文件，处理外设中断事件
  - `stm32fxxx_hal_conf.h`：HAL库配置文件，可裁剪不需要的外设驱动
- **开发建议**：大部分业务逻辑代码都应写在此目录，与底层驱动解耦。

## 目录分层逻辑
本工程采用了清晰的分层设计，从底层到上层依次为：
`Drivers（硬件驱动）` → `Middlewares（中间件）` → `User（应用代码）`
这种结构的优势：
1.  **解耦**：底层驱动与业务代码分离，更换芯片或外设时，只需修改`Drivers`目录，无需改动上层逻辑。
2.  **可维护性强**：文件分类清晰，便于后续扩展与调试。
3.  **Git友好**：编译产物、工程配置文件可单独忽略，提交历史更干净。
